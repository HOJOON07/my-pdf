# UX/UI 설계 - PDF 회전(Rotate) / 순서 바꾸기(Reorder) 화면

> 버전: 1.1
> 기준 문서: PRD v1.2, PDF_ROTATE_REORDER_SPEC.md
> 작성일: 2026-03-19
> 기존 Merge / Split / Extract / Delete UI 패턴과 일관성 유지

---

## 1. 정보 구조 (IA)

```
vibe-pdf
├── 병합 (Merge)          ← 기존
├── 분할 (Split)          ← 기존
├── 추출 (Extract)        ← 기존
├── 삭제 (Delete)         ← 기존
├── 회전 (Rotate)         ← 신규
└── 순서 바꾸기 (Reorder) ← 신규
```

탭이 6개로 늘어나므로 탭 바 레이아웃 조정 필요 (아래 섹션 5.0 참고).

## 2. 기능 정의

### Rotate (페이지 회전)
- 특정 페이지(또는 전체)를 90 / 180 / 270도(시계 방향) 회전하여 새 PDF 저장
- **범위 미입력 = 전체 페이지 적용** (PRD R-04)
- 회전 각도 선택 없이 실행 불가 (AC-R-06)
- 출력 파일명: 기본값 `{원본파일명}_rotated.pdf`, 사용자 변경 가능
- 기존 페이지 회전값에 누적 적용 (스펙 2.2)

### Reorder (페이지 순서 바꾸기)
- 페이지 번호가 표시된 카드 목록을 드래그앤드롭으로 순서 재배열
- Reset 버튼으로 원본 순서 복구 가능 (RO-10)
- 1페이지 PDF는 드래그 비활성화, 안내 메시지 표시 (E-RO-01)
- 북마크(목차)가 있는 PDF의 경우 북마크 손실 고지 (스펙 4.2)
- 출력 파일명: 기본값 `{원본파일명}_reordered.pdf`, 사용자 변경 가능

## 4. 주요 사용자 플로우

### 4.1 Rotate 플로우

```
[초기] 파일 없음
  → FileDropZone 표시 (파일 1개)
  → 파일 드롭 or 클릭 선택
[파일 로드됨]
  → 파일 정보 카드 (파일명, 전체 페이지 수)
  → 페이지 범위 입력 (비워두면 전체 페이지 적용)
  → 회전 각도 선택: 90도 / 180도 / 270도 라디오 버튼
  → 출력 파일명 (기본값: {원본}_rotated.pdf)
  → 각도가 선택되면 "PDF 회전하기 →" 버튼 활성화
  → 버튼 클릭
[처리 중]
  → 스피너 + "회전 중...", 모든 입력 disabled
[완료]
  → 성공 카드 (초록 배경, CheckCircle 아이콘)
  → "새 작업 시작" 버튼
[오류]
  → 에러 배너 + 입력 상태 유지, 재시도 가능
```

### 4.2 Reorder 플로우

```
[초기] 파일 없음
  → FileDropZone 표시 (파일 1개)
  → 파일 드롭 or 클릭 선택
[파일 로드됨]
  → 파일 정보 카드 (파일명, 전체 페이지 수)
  → 페이지 카드 목록 (각 카드에 "1", "2", ... 페이지 번호 표시)
  → 드래그앤드롭으로 카드 순서 변경
  → (1페이지 PDF인 경우 드래그 비활성 + 안내)
  → Reset 버튼으로 원본 순서로 복구
  → 출력 파일명 (기본값: {원본}_reordered.pdf)
  → "저장하기 →" 버튼 (항상 활성, 순서 변경 없어도 허용)
  → 버튼 클릭
[처리 중]
  → 스피너 + "처리 중...", 모든 입력 disabled
[완료]
  → 성공 카드 (초록 배경, CheckCircle 아이콘)
  → "새 작업 시작" 버튼
[오류]
  → 에러 배너 + 입력 상태 유지
```

## 5. 레이아웃 설명

### 5.0 App.tsx 탭 바 (변경)

탭이 6개로 늘어나므로 기존 `px-4` padding을 유지하면 800px 내에서 수평 스크롤이 발생할 수 있다.
대응 방안: 탭 바에 `overflow-x-auto` 적용 + 스크롤바 숨김으로 가로 스크롤 허용.
탭 라벨은 한국어 짧게 유지.

```
┌────────────────────────────────────────────────────────────────────┐
│  vibe-pdf                          [파일이 내 기기를 떠나지 않아요]   │
├────────────────────────────────────────────────────────────────────┤
│  병합 │ 분할 │ 추출 │ 삭제 │ 회전 │ 순서 바꾸기                       │
│  (overflow-x-auto, scrollbar-hidden)                               │
└────────────────────────────────────────────────────────────────────┘
```

탭 타입 변경:
```typescript
type Tab = 'merge' | 'split' | 'extract' | 'delete' | 'rotate' | 'reorder'

const tabs = [
  { id: 'merge',   label: '병합 (Merge)' },
  { id: 'split',   label: '분할 (Split)' },
  { id: 'extract', label: '추출 (Extract)' },
  { id: 'delete',  label: '삭제 (Delete)' },
  { id: 'rotate',  label: '회전 (Rotate)' },
  { id: 'reorder', label: '순서 바꾸기 (Reorder)' },
]
```

탭 바 클래스 변경: `flex border-b border-gray-200 overflow-x-auto scrollbar-hide`

## 7. RotateDegreeSelector 컴포넌트 상세

SplitModeSelector 패턴을 따른다 (라디오 버튼 목록).

### Props

```typescript
interface RotateDegreeSelectorProps {
  value: 90 | 180 | 270 | null  // null = 미선택
  onChange: (deg: 90 | 180 | 270) => void
  disabled?: boolean
}
```

### 옵션 목록

| 값 | 레이블 | 아이콘 |
|----|--------|--------|
| 90 | 90도 시계 방향 | RotateCw (lucide-react) |
| 180 | 180도 (뒤집기) | RefreshCw (lucide-react) |
| 270 | 270도 시계 방향 (반시계 90도) | RotateCcw (lucide-react) |

### 레이아웃 구조

```
<fieldset>
  <legend>회전 각도 선택</legend>
  [라디오 옵션 1: 90도]  ← border-blue-500 bg-blue-50 (선택 시)
  [라디오 옵션 2: 180도]
  [라디오 옵션 3: 270도]
</fieldset>
```

SplitModeSelector와 동일 패턴:
```
className={cn(
  'flex cursor-pointer items-center gap-3 rounded-md border px-4 py-3 transition-colors',
  value === opt.value ? 'border-blue-500 bg-blue-50' : 'border-gray-200 bg-white hover:bg-gray-50',
  disabled && 'cursor-not-allowed opacity-50'
)}
```

각도 선택 옵션에 아이콘을 추가하여 시각적 명확성 제공.

## 8. PageCardList / PageCard 컴포넌트 상세 (Reorder)

기존 `FileList` + `FileListItem` (dnd-kit)의 패턴을 재사용하되,
파일 목록이 아닌 페이지 카드 목록으로 변형한다.

### PageCard props

```typescript
interface PageCardProps {
  /** 원본 페이지 번호 (1-based, 표시용) */
  originalPageNumber: number
  /** 드래그 식별자 (useSortable용) */
  id: string
  disabled?: boolean
}
```

### 카드 구조

```
[GripVertical 핸들]  [  N  ]  N페이지
```

- `[GripVertical]`: 드래그 핸들 (기존 FileListItem 패턴 동일)
- `[ N ]`: 원본 페이지 번호 배지 (회색 배경 rounded)
- `N페이지`: 텍스트 레이블

X 버튼 없음 — 페이지 제거 기능은 Reorder 범위 밖

### 카드 클래스

기존 FileListItem과 동일:
```
flex items-center gap-3 rounded-md border border-gray-200 bg-white px-3 py-2.5
```

드래그 중: `opacity-50 shadow-lg`

## 9. 훅 상태 구조

### 9.1 useRotate

```typescript
type RotateDegree = 90 | 180 | 270

interface RotateState {
  file: { file: File; pageCount: number } | null
  rangeInput: string             // 비어있으면 전체 페이지 적용
  rangeError: string | null
  degree: RotateDegree | null    // 미선택 시 null
  outputName: string             // 기본값: '{원본}_rotated.pdf'
  status: ProcessingStatus
  error: string | null
  canExecute: boolean            // file !== null && degree !== null && rangeError === null
}
```

`canExecute` 조건: 파일 있음 AND 각도 선택됨 AND 범위 오류 없음
범위가 비어 있어도 유효 (전체 페이지 적용)

### 9.2 useReorder

```typescript
interface PageItem {
  id: string              // dnd-kit 식별자 (고정 uuid)
  originalIndex: number   // 원본 0-based 인덱스 (불변)
  originalPageNumber: number  // 원본 1-based 번호 (표시용)
}

interface ReorderState {
  file: { file: File; pageCount: number } | null
  pages: PageItem[]          // 현재 순서 (드래그로 변경됨)
  outputName: string         // 기본값: '{원본}_reordered.pdf'
  status: ProcessingStatus
  error: string | null
  canExecute: boolean        // file !== null && pageCount >= 2
}
```

`pages` 배열의 순서가 새 PDF의 페이지 순서를 결정한다.
실행 시 `pages.map(p => p.originalIndex)`를 `reorderPages(file, orderedIndices)`에 전달.

## 13. 접근성 / 사용성 고려사항

1. **탭 순서 (Tab key)**
   - Rotate: FileDropZone → X버튼 → 범위 입력 → 각도 라디오 그룹 → 출력 파일명 → 실행 버튼
   - Reorder: FileDropZone → X버튼 → 페이지 카드 드래그 핸들들 → Reset 버튼 → 출력 파일명 → 저장 버튼

2. **RotateDegreeSelector 접근성**
   - `<fieldset>` + `<legend>` 구조 (SplitModeSelector와 동일)
   - `name="rotate-degree"` 라디오 그룹으로 묶어 키보드 화살표 키 탐색 지원

3. **PageCardList (Reorder) 접근성**
   - dnd-kit의 키보드 접근성 지원: Space로 드래그 시작/종료, 화살표 키로 이동
   - `aria-label="페이지 {N} 순서 변경"` 각 드래그 핸들에 적용

4. **aria-label**: 파일 제거 버튼 `${file.name} 제거` — 기존 패턴 동일

5. **role="alert"**: 처리 오류 배너

6. **role="status"**: Reorder 1페이지 안내 박스 (오류가 아닌 정보성 메시지)

7. **disabled 상태**: 처리 중 모든 인터랙티브 요소 `disabled` + `opacity-50`

8. **색상 대비**: WCAG AA 기준 충족
   - 파란 정보 박스: `text-blue-800` on `bg-blue-50`
   - 빨간 에러: `text-red-700` on `bg-red-50`

9. **모바일 (375px)**
   - 탭 바: 가로 스크롤로 접근 가능
   - 페이지 카드: 터치 드래그앤드롭 지원 (dnd-kit 기본 지원)
   - RotateDegreeSelector: 세로 스택 레이아웃 (기존 SplitModeSelector와 동일)

## 16. Architect 확인 사항 응답

Frontend Architect가 요청한 3가지 사항에 대한 결정 및 근거.

### Q1. 탭 6개 overflow 처리 방식

**결정: 가로 스크롤 (overflow-x-auto + scrollbar-hide)**

```
nav 클래스: flex border-b border-gray-200 overflow-x-auto scrollbar-hide
```

**드롭다운, 2줄 배치를 선택하지 않은 이유:**

| 방식 | 검토 결과 |
|------|----------|
| 드롭다운 | 탭 UI 패턴을 벗어나 인지 일관성이 깨짐. 현재 기능이 6개 고정이므로 과도한 설계 |
| 2줄 배치 | 헤더 영역이 늘어나 컨텐츠 영역이 줄어듦. 800px max-width 기준에서 레이아웃 불안정 |
| 가로 스크롤 | 기존 탭 구조 변경 없음. 모바일에서 터치 스크롤로 자연스럽게 동작. scrollbar-hide로 데스크톱에서 스크롤바 비표시 |

**구현 방법**: Tailwind의 `scrollbar-hide` 유틸리티(tailwind-scrollbar-hide 플러그인 또는 CSS `scrollbar-width: none`) 사용.

**데스크톱 800px에서 실제 overflow 발생 여부**: 6개 탭 라벨 합산 예상 폭 약 650~700px로 `px-4 py-2.5` 기준 800px 내 수용 가능. overflow-x-auto는 보험 처리.

### Q2. Reorder 순서 미변경 시 저장 버튼 활성화 여부

**결정: 항상 활성화 (순서 미변경 시에도 버튼 활성)**

**근거: PRD AC-RO-06 명시**

> AC-RO-06: 순서를 변경하지 않은 상태(원본과 동일)에서 저장 버튼을 클릭하면 원본과 동일한 PDF가 다운로드된다.

PRD에서 미변경 상태의 저장을 명시적으로 허용하고 있다. 버튼을 비활성화하면 이 AC를 위반한다.

**UX 근거:**
- 사용자가 파일을 업로드하고 "순서가 이미 맞다"고 판단하여 그대로 저장하는 시나리오가 유효하다
- 변경 여부를 판단해서 버튼 상태를 바꾸면 "변경을 했는데 버튼이 왜 비활성이지?"라는 혼란이 생길 수 있다
- `canExecute: file !== null && pageCount >= 2` — 파일 존재 + 2페이지 이상이면 항상 활성

**Architect 권장안 불채택 이유**: PRD AC-RO-06와 직접 충돌. PRD가 기준 문서이므로 우선한다.

---

### Q3. Rotate PageRangeInput 빈값 안내 방법

**결정: placeholder + 상시 힌트 텍스트 2중 안내**

범위 입력 필드에 두 가지 위치에서 동시에 안내한다.

```
레이블:       회전할 페이지 (전체: {N}페이지)
placeholder:  예: 1, 3-5  (비워두면 전체 페이지에 적용)
힌트 텍스트:  비워두면 전체 페이지에 회전이 적용됩니다
```

**placeholder 단독으로 충분하지 않은 이유:**
- placeholder는 입력 중에 사라짐 — 사용자가 이미 타이핑을 시작했다면 안내를 놓칠 수 있다
- 힌트 텍스트는 항상 표시되므로 필드를 비우고 실행하기 직전에도 확인 가능

**타 기능과의 차이:**
- Extract/Delete: 범위 미입력 = 버튼 비활성 → 힌트 없어도 자연스럽게 입력을 유도함
- Rotate: 범위 미입력 = **유효한 상태 (전체 적용)** → 의도적 빈값을 명확히 안내해야 함

**PRD 근거**: R-04 "비워두면 전체 페이지 적용", E-R-01 동일.

**구현 방법**: PageRangeInput의 기존 힌트 텍스트 슬롯(id="range-hint")을 그대로 활용, Rotate에서는 "비워두면 전체 페이지에 회전이 적용됩니다" 문구를 전달.
